# 문제 해결 원칙 (Legacy)

> 이 문서는 레거시 스킬 호환을 위해 유지됩니다.
> 현재 Active 기준 문서는 `docs/skill-ops/framework.md`와 각 스킬의 `references/philosophy.md`입니다.

**최종 업데이트:** 2026-02-26 (4차)
**누적 문제 수:** 3개
**상태:** 사례 연구 리서치 단계 추가 (Case Study Research)

---

## 1. 핵심 원칙: 2W1H

### 순서의 중요성
대부분의 사람들은 **How(어떻게)**부터 생각합니다.
What(무엇)과 Why(왜)는 남이 정해줍니다.

문제 해결은 **역순**으로 시작해야 합니다:

1. **Why (왜)**: 왜 이 문제를 해결해야 하는가?
   - 산업/맥락 이해
   - 배경 지식 습득
   - 진짜 목적 파악

2. **What (무엇)**: 무엇이 문제인가?
   - 문제 정의
   - 모호한 용어 해체
   - 경계 설정

3. **How (어떻게)**: 어떻게 해결할 것인가?
   - 방법론 선택
   - 기술 스택 결정
   - 구현 전략

### ⚠️ How부터 시작하면 생기는 일
- 방향을 잃음
- 불필요한 기술 도입
- 문제 재정의 → 로드맵 전체 수정

**실제 사례:**
"대규모 데이터 처리를 위해 MSA로 시작" → MSA는 도메인 분리지 대규모 데이터 처리가 아님 → 전체 재정의

## 2. 질문 템플릿 워크플로우

### 원칙
2W1H를 실천하기 위한 구체적 도구로 **질문 템플릿 (questions.md)** 를 사용한다.

**문제:** 터미널 대화는 가독성이 떨어지고, 질의응답이 문서로 남지 않는다.
**해결:** 템플릿 파일을 제공하고, 사용자가 에디터에서 작성하게 한다.

### 워크플로우

1. **사용자가 자연어로 문제 제기**
   - 예: "애자일 코치 도구를 만들고 싶어"

2. **AI가 questions.md 생성**
   - `templates/questions.md`를 `problems/[문제명]/questions.md`로 복사
   - 사용자에게 알림: "질문지를 생성했습니다. 에디터에서 편하게 작성해주세요."

3. **사용자가 에디터에서 작성**
   - 4개 질문 (What, Why, 제약조건, 우선순위)
   - 2W1H 프레임워크 강제

4. **사용자가 완료 신호**
   - "질문지 작성 완료했어. 이제 브레인스토밍 계속하자."

5. **AI가 questions.md 읽고 분석**
   - What/Why 재정의
   - 모호한 용어 해체
   - 산업 리서치 필요 시 딥 리서치 프롬프트 제안

6. **브레인스토밍 계속 + 2w-brainstorm.md 생성**
   - questions.md: 사용자 답변 저장
   - 2w-brainstorm.md: AI와의 추가 대화 기록

### 질문 템플릿 구조

**`templates/questions.md` 구조:**

```markdown
## 📋 What (무엇) - 문제 정의
Q1: 현재 상황과 불편함

## 🤔 Why (왜) - 목적
Q2: 해결 후 달라질 점

## ⏱️ How (어떻게) - 제약 조건
Q3: 시간과 범위

## 🎯 우선순위
Q4: 가장 중요한 것
```

**핵심:**
- 질문 개수: 4개 (14개 → 4개로 축소)
- 밀도: 높음 (예시 제공, 2W1H 구조화)
- AI 추론: 부족한 정보는 AI가 읽고 추가 질문

### 장점

✅ **문서 자동 저장:** questions.md 자체가 기록
✅ **에디터에서 편한 작성:** 터미널 불편함 해소
✅ **가독성 좋음:** 긴 질문을 md 파일에서 읽음
✅ **2W1H 프레임워크 강제:** 템플릿 자체가 프레임워크

### 고수/초보 구분 불필요

**초반 고민:** "질문을 초보용/고수용으로 나눌까?"

**결론:** 불필요. 질문 템플릿 자체가 프레임워크를 가르치므로:
- 초보: 템플릿이 사고를 유도
- 고수: 템플릿이 구조화 도와줌

**핵심 깨달음:**
> "고수/초보"가 아니라, **"프레임워크를 따르는가/안 따르는가"**가 중요하다.

## 3. 사고 패턴: 보편 ↔ 구체 순환

### 모르는 분야 (Top-Down)
**보편적 개념 → 구체적 사례**

1. 산업 전반 탐색 (Deep Research)
2. 이론/패턴 학습
3. 구체적 사례 적용
4. 요구사항 도출

**예시:**
- 핀테크를 모름 → 핀테크 산업 리서치 → 페인 포인트 파악 → 우리 서비스 요구사항 도출

### 아는 분야 (Bottom-Up)
**구체적 사례 → 보편적 개념**

1. 현상/버그 관찰
2. 패턴 발견
3. 공통 본질 추출
4. 일반화된 해결책

**예시:**
- 코드에서 반복되는 문제 발견 → 패턴 분석 → 디자인 패턴 적용 → 리팩토링

### 핵심
**"방향을 전환하는 순간"이 breakthrough입니다.**
- 보편 → 구체로 내려가다 막히면
- 구체적 사례들을 모아 보편 재정의

## 4. 문제 공간 vs 해결 공간 (DDD/SoC)

### 문제 공간 (Problem Space)
**What/Why 영역 - 도메인**

- 무엇을 해결할 것인가?
- 왜 해결해야 하는가?
- 비즈니스 규칙은 무엇인가?

**도구:**
- 산업 리서치
- 고객 인터뷰
- 도메인 모델링
- 유비쿼터스 언어

**중요:** 기술을 고민하지 않습니다.

### 해결 공간 (Solution Space)
**How 영역 - 기술**

- 어떻게 구현할 것인가?
- 어떤 기술 스택을 쓸 것인가?
- 아키텍처는 어떻게 할 것인가?

**도구:**
- 디자인 패턴
- 프레임워크
- 인프라
- 아키텍처

### 관심사의 분리 (SoC)
**문제 공간과 해결 공간을 명확히 분리합니다.**

- ❌ 동시에 고민: "도메인이 뭐지... 근데 Kafka를 써야 하나?"
- ✅ 순차적 고민: "도메인이 명확해졌다 → 이제 기술 선택"

---

## 5. 3단계 프레임워크

### Phase 1: Discovery (문제 정의)
1. **산업 탐색**: 거시적 시장/고객 페인 포인트
2. **가설 수립**: "이것이 진짜 문제일 것이다"
3. **갭 보정**: 이론과 실무의 차이, 공통 언어 정의

### Phase 2: Design (솔루션 설계)
1. **방법론 선정**: 조직/예산/역량 고려 (DDD, 애자일 등)
2. **도메인 모델링**: 비즈니스 규칙 구조화
3. **기술 아키텍처**: Build vs Buy 결정

### Phase 3: Delivery (실행 및 검증)
1. **PoC/MVP**: 핵심 가치만 담은 최소 기능
2. **Production**: 안정성/확장성 고려 본 개발
3. **회고 및 측정**: 비즈니스 지표 확인 → Phase 1로 순환

## 6. 흔한 실수 패턴

### 모호한 용어 그대로 사용
❌ "대규모 데이터 처리" → 뭉뚱그린 개념
✅ 해체 필요:
- 경쟁 상태 (동시성)?
- 빅데이터 (용량)?
- 조회 최적화 (성능)?
- 이벤트 처리 (EDA)?

### What 없이 How부터 시작
❌ "MSA로 만들어야지" → 왜? 무엇을?
✅ "이 시스템이 해결할 문제는 뭐지?" → MSA가 정말 필요한가?

### 문제 공간과 해결 공간 혼재
❌ 도메인 회의에서 "DB 어떻게 할까요?" 논의
✅ 도메인 먼저 → 기술은 나중에

### 프로젝트 성격 혼동
❌ **목적과 형태 불일치:**
- 토이 프로젝트인데 MVP 수준 범위
- 이직용인데 비즈니스 완성도 추구
- 학습용인데 Production 수준 인프라

✅ **목적에 맞는 형태:**

| 목적 | 집중할 것 | 생략할 것 |
|------|----------|----------|
| 기술 증명 (PoC) | 정량 지표, 비교 분석 | 완전한 비즈니스 로직 |
| 학습 | 개념 이해, 회고 | 완성도, 배포 |
| 이직용 | 블로그, 재현 가능성 | 실제 사용자, UX |
| 실제 서비스 (MVP) | 핵심 가치, 사용자 검증 | 완벽한 기능 |

**실제 사례:**
- 이직용 프로젝트에 "회원가입/장바구니/관리자" 구현 → 불필요
- PoC에 "헥사고날 + DDD + Kafka" → 과도
- 학습용에 "CI/CD + K8s 배포" → 본질과 무관

## 7. 제약 조건 기반 범위 산정

### 원칙
What/Why를 정의한 후, **제약 조건**을 먼저 확인하고 범위를 결정한다.

**핵심:** 아무리 좋은 아이디어라도, 제약 조건을 무시하면 실패한다.

### 제약 조건 체크리스트

문제 정의 직후 반드시 확인:

1. **시간:** 얼마나 시간을 쓸 것인가?
   - 1주? 1달? 3달? 6달?
   - 데드라인이 있는가?

2. **목적:** 왜 만드는가?
   - 비즈니스 수요? (실제 사용자)
   - 학습? (개념 이해)
   - 이직용 증명? (정량 지표)
   - 연구? (가설 검증)

3. **협업:** 누가 하는가?
   - 혼자? 팀?
   - 온보딩이 필요한가?
   - 스터디/블로그 공유?

4. **완성도:** 어디까지 만들 것인가?
   - PoC (개념 증명)?
   - MVP (최소 기능)?
   - Production (실제 서비스)?

### 범위 결정 매트릭스

| 목적 | 시간 | 적합한 형태 | 핵심 산출물 | 예시 |
|------|------|------------|-----------|------|
| **기술 증명** | 1-2달 | **PoC** | 정량 지표, 비교 분석 | 동시성 제어 4가지 비교 |
| **학습** | 1-3달 | **Tutorial** | 학습 노트, 회고 | Spring WebFlux 따라하기 |
| **이직용** | 1-2달 | **PoC + 블로그** | GitHub, 블로그, 재현 가능 | 재고 차감 성능 비교 |
| **실제 서비스** | 3-6달+ | **MVP** | 사용자, 피드백 | 최소 기능 커머스 |
| **연구** | 3-6달 | **Experiment** | 논문, 데이터셋 | 알고리즘 성능 실험 |

### 흔한 실수

❌ **제약 무시:**
- "1-2달 제약인데 4개 Sprint 계획 (3-4달 소요)"
- "토이 프로젝트인데 MVP 수준 범위 (3개 도메인, 헥사고날, Kafka)"
- "혼자 하는데 팀 프로젝트 수준 기획"

✅ **제약 기반 결정:**
- "1-2달 → PoC 범위"
- "토이 프로젝트 → 하나를 깊게"
- "혼자 → 아키텍처 단순화"

### 범위 축소 기준

**다음 중 하나라도 해당하면 범위 축소:**
1. 예상 시간이 제약의 1.5배 이상
2. "나중에 추가" 항목이 3개 이상
3. 완성 가능성에 확신이 없음
4. 핵심 기능과 무관한 기능이 30% 이상

### 실제 사례

**"대규모 트래픽 처리 이직용 프로젝트"**

❌ **제약 무시:**
- 시간: 1-2달
- 계획: time-deal-service (3개 도메인 + 헥사고날 + Kafka)
- 예상: 3-4달 소요
- 결과: 미완성 위험

✅ **제약 기반:**
- 시간: 1-2달
- 계획: 동시성 제어 PoC (4가지 Lock 비교)
- 예상: 1-1.5달
- 결과: 완성 + 정량 지표 + 블로그

## 8. 산업 리서치 프롬프트

### 원칙
What을 정의할 때, **가설이 아닌 실제 데이터**를 기반으로 한다.

**문제:** AI에게 WebSearch를 시키면 토큰 소모가 큼
**해결:** 사용자가 직접 리서치할 수 있는 **딥 리서치 프롬프트**를 제안

### 리서치가 필요한 신호

**다음과 같은 말이 나오면 리서치 필요:**
- "~일 것 같아" (가설)
- "~가 중요할 거야" (추측)
- "~면 좋을 것 같아" (희망)
- "대규모 트래픽 처리" (모호한 용어)

**대응:**
→ "그 가설의 근거가 있나요?"
→ "딥 리서치 프롬프트를 제안해드릴까요?"

### 딥 리서치 프롬프트 템플릿

AI는 다음 형식으로 프롬프트를 제안:

```
📋 [주제] 딥 리서치 프롬프트

다음 내용을 ChatGPT/Claude/Perplexity에 붙여넣고 리서치하세요:

---

[상황 설명]
나는 [목적]을 위해 [문제]를 해결하려고 합니다.

[리서치 질문]
1. [타겟 회사/산업]에서 "[키워드]"의 실제 의미는 무엇인가요?
2. 관련 기술 블로그 5개를 찾아 공통적으로 다루는 문제를 정리해주세요.
3. 채용 공고(JD) 10개를 분석해서 자주 등장하는 기술 키워드를 추출해주세요.
4. "[키워드]"와 유사하지만 다른 개념(예: X vs Y)을 비교 설명해주세요.

[출력 형식]
- 실제 의미: ...
- 공통 문제: ...
- 핵심 키워드: ...
- 개념 비교: ...
- 추천 학습 우선순위: ...

---

리서치 완료 후, 결과를 바탕으로 다시 논의하겠습니다.
```

### 실제 예시

**상황:** "대규모 트래픽 처리" 정의 필요

**AI 제안 프롬프트:**
```
📋 대규모 트래픽 처리 딥 리서치 프롬프트

다음 내용을 ChatGPT/Perplexity에 붙여넣고 리서치하세요:

---

나는 이직용 포트폴리오를 위해 "대규모 트래픽 처리" 기술을 증명하려고 합니다.
타겟 회사는 네카라쿠배, 뱅크샐러드, 토스 등입니다.
```

## 9. 사례 연구 리서치 (Case Study Research)

### 원칙
How를 구조화하기 전에, **"선배들은 언제/왜 이 기술을 도입했는가"**를 사례로 학습한다.

**핵심:**
> 내 지식만으로는 "도입 맥락"이 부족하다. 기술 블로그의 사례에는 **도입 배경, 트레이드오프, 실패 경험**이 담겨 있다.
> 이 인사이트가 다이어그램 설계와 평가 지표 선정에 직접 영향을 준다.

### 사례 연구가 필요한 시점

**2W(What/Why) 명확화 후, 1H(How 구조화) 전에 수행한다.**

```
[2W1H Pipeline에서의 위치]
/2w-brainstorm → What/Why 정리
        ↓
   사례 연구 리서치 ← 여기 (2W와 1H 사이)
        ↓
/1h-agile-phase → 다이어그램 + Phase + 지표
```

### 필요 여부 판단

| 상황 | 필요? | 이유 |
|------|-------|------|
| 처음 다루는 기술 | ✅ 필수 | 도입 맥락 없이 How를 구조화하면 가설에 의존 |
| 익숙한 기술의 새로운 적용 | ⚠️ 선택 | 부족한 부분만 보완 |
| 이미 충분히 리서치 | ❌ 불필요 | 중복 방지 |

### 수집할 인사이트

1. **도입 배경:** 어떤 문제 때문에 이 기술을 도입했는가?
2. **트레이드오프:** 도입 후 겪은 어려움은?
3. **핵심 시나리오:** PoC에서 가장 임팩트 있는 테스트는?

### 1H에 전달하는 방식

사례 연구 인사이트는 `2w-brainstorm.md`에 기록하고, `/1h-agile-phase`에서 다음에 반영:
- **다이어그램:** 사례에서 발견한 아키텍처 패턴
- **평가 지표:** 사례에서 핵심이었던 측정 항목
- **In/Out/Deferred:** 사례 기반 범위 판단

### 실제 사례

**EDA PoC (2026-02-26):**
- 사례 연구 **없이** 진행 → how-diagram.md 작성 후 "장애 격리 테스트"라는 인사이트 발견
- 사례 연구를 **먼저** 했다면 → "배민/토스에서 EDA 도입 시 장애 격리가 핵심 동기"를 미리 파악
- **교훈:** 사례 연구 인사이트가 평가 지표 설계에 직접 영향 → 2W 단계에서 수집해야 함

## 10. 완성 가능성 우선

### 원칙
**미완성 복잡한 프로젝트 < 완성된 단순 프로젝트**

이직/학습 목적이라면 **"완성했다"**는 것 자체가 가치다.

### 완성의 정의

**완성된 프로젝트:**
- ✅ 정량 지표 측정 (TPS, Latency, Before/After)
- ✅ 재현 가능 (README만 보고 실행 가능)
- ✅ 문서화/블로그 (Why/What/How 명확)
- ✅ 명확한 결론 ("어떤 상황에 어떤 방법")

**미완성 프로젝트:**
- ❌ "구현 중입니다"
- ❌ "나중에 추가할 예정"
- ❌ 복잡한 구조만 있고 측정 없음
- ❌ "이론적으로 좋다"만 있고 실제 데이터 없음

### 완성 가능성 체크

**시작 전 필수 질문 3가지:**

1. **정량 측정 가능한가?**
   - "더 빠르다" (X) → "TPS가 2배 증가" (O)
   - "성능이 좋다" (X) → "Latency p95가 50ms → 20ms" (O)

2. **다른 사람이 재현 가능한가?**
   - Docker Compose로 5분 안에 실행?
   - README 보고 따라할 수 있나?
   - 환경 차이로 안 될 가능성?

3. **명확한 결론이 있는가?**
   - "A가 B보다 좋다" (결론 있음)
   - "여러 방법을 시도했다" (결론 없음)

**하나라도 불확실하면 범위 축소**

### 완성도 vs 완성 가능성

| 구분 | 완성도 높음 | 완성 가능성 높음 |
|------|-----------|---------------|
| 범위 | 넓고 깊음 | 좁고 깊음 |
| 기간 | 3-6달 | 1-2달 |
| 위험 | 미완성 가능 | 완성 보장 |
| 결과 | "만들다 말았어요" | "이렇게 측정했어요" |

**이직/학습용이라면 완성 가능성을 선택**

### 완성 우선 전략

**1단계: 최소 범위로 시작**
- 가장 핵심 기능 하나만
- 정량 측정 가능한 것만
- 1-2주 안에 완성 가능한 것만

**2단계: 완성 + 측정**
- 정량 지표 측정
- 블로그 작성
- GitHub 공개

**3단계: 확장 여부 결정**
- 완성 후 시간 남으면 확장
- 미완성 상태에서 확장 금지

### 실제 사례

**"동시성 제어 PoC"**

❌ **완성도 추구 (미완성 위험):**
- 범위: 3개 도메인 + 4가지 기술 + 헥사고날
- 예상: 3-4달
- 위험: 각각이 얕거나 미완성

✅ **완성 가능성 우선:**
- 범위: 1개 도메인 + 4가지 Lock 비교
- 예상: 1-1.5달
- 결과: 완성 + 정량 측정 + 블로그
- 확장: 완성 후 시간 있으면 2번째 PoC

## 12. 계획 MVP 원칙 (Planning as MVP)

> "초기 계획이 바뀌지 않으면 그게 더 문제다. 계획 변경은 학습의 증거다."
> — eda-poc 회고 (2026-03-01)

### 핵심 명제

| 잘못된 믿음 | 올바른 이해 |
|------------|------------|
| 초기 계획이 바뀌면 계획이 나빴던 것 | 초기 계획은 **제한된 정보 하의 최선** |
| 계획 변경 = 실패 | 계획 변경 = 학습의 증거 |
| 계획이 많이 바뀔수록 문제 | 계획이 안 바뀌면 학습이 없었다는 신호 |

### 2w+1h 산출물의 역할

2w-brainstorm.md와 how-diagram.md는 **완성된 계획이 아니라 Planning MVP**다.

- **뼈대/기준점**으로 작동 → 학습 방향을 잡아줌
- 뼈대 없이 학습 → 지식이 파편화됨
- 뼈대 있이 학습 → 학습이 **계획 업그레이드**로 연결됨

### AI 시대의 계획

- 예전: 계획 한 번 = 몇 주 회의 → 변경 비용이 너무 컸음
- 지금: AI와 함께 빠르게 iteration → **계획 자체에 애자일 적용 가능**

### 실사례

**eda-poc (2026-03-01):**
2w+1h 완료 후 부족한 개념(EDA, Kafka, Avro 등)을 직접 학습.
학습을 통해 how-diagram.md를 더 나은 계획으로 업그레이드.
결론: 초기 계획이 MVP → 학습 → 계획 업그레이드의 선순환 확인.
