# 2W Problem Definition (What/Why)

## 목적
새로운 문제를 만났을 때 How(어떻게)부터 시작하지 않고, **What(무엇)과 Why(왜)를 먼저 명확히** 하는 브레인스토밍 프로세스를 진행합니다.

**이 스킬의 범위:**
- 2W(What/Why) 정의 + 제약조건 확인
- How는 `/1h-agile-phase` 스킬에서 진행
- (Optional) How 초안 제안 가능

## 전체 파이프라인

```
[2W1H Pipeline]
/2w-brainstorm → What/Why 정리 + (optional) How 초안 ← 현재 스킬
        ↓
/1h-agile-phase → 다이어그램(범위) + Phase(단계) + 지표(평가)
```

**관심사의 분리:**
- 2W만 필요할 때 → `/2w-brainstorm`만 사용
- 1H만 필요할 때 → `/1h-agile-phase`만 사용
- 전체 흐름 → 2W → 1H 순서로 진행

## 필수 사전 준비

**작업 시작 전 반드시 읽을 것:**
- 📖 `problem-solving-principles.md` **(전체 읽기 필수!)**

특히 다음 섹션 주의:
- "2W1H 원칙" - 순서의 중요성
- "How부터 시작하면 생기는 일" - 함정 사례
- "흔한 실수 패턴" - 모호한 용어 해체

### 브레인스토밍 중 (2W)
- [ ] What이 명확한가? (모호한 용어 해체했나?)
- [ ] Why가 명확한가? (왜 해야 하는지 이해했나?)
- [ ] **제약 조건** 확인했나? (시간/협업/완성도)
- [ ] **목적과 형태** 일치하나? (PoC/MVP/Tutorial)
- [ ] 산업 리서치 필요하면 **딥 리서치 프롬프트** 제안했나?
- [ ] **사례 연구 프롬프트** 제안했나? (선배들은 언제/왜 도입했는가?)
- [ ] How 이야기가 나와도 일단 보류하고 있나?

### 1H로 넘어가기 전 체크 (2W 완료 조건)
- [ ] 범위가 제약 조건 내에서 **완성 가능**한가?
- [ ] "완성"의 정의가 명확한가? (정량 지표/재현 가능/문서화/결론)
- [ ] **사례 연구 인사이트**가 수집되었나? (또는 불필요 판단)
- [ ] 패턴 감지 체크리스트 3개 이상 통과했나?
- [ ] **→ 통과하면 `/1h-agile-phase`으로 이동 안내**

## 작동 방식 (Workflow)

### 1. 문제 브레인스토밍 시작

**Step 1: How 함정 감지**
사용자가 이런 말을 하면 **즉시 중단**:
- "MSA로 시작하려고..."
- "Kafka를 사용해서..."
- "Redis로 캐싱하고..."

→ **"잠깐만요! 2W1H 브레인스토밍을 시작할게요"** 선언

**Step 2: 준비 및 questions.md 생성**
- `problem-solving-principles.md` 읽기
- 문제 이름 확인
- `problems/[문제명]/` 디렉토리 생성
- **`templates/questions.md`를 `problems/[문제명]/questions.md`로 복사**
- 사용자에게 알림: "질문지를 생성했습니다. 에디터에서 편하게 작성해주세요."

### 2. questions.md 작성 대기

**중요: 터미널에서 대화로 질문하지 않고, questions.md 파일을 제공**

**워크플로우:**
1. AI가 `problems/[문제명]/questions.md` 생성 (템플릿 복사)
2. 사용자가 에디터에서 작성
3. 사용자가 "질문지 작성 완료" 신호
4. AI가 `questions.md` 읽기
5. 답변 기반으로 브레인스토밍 계속

**장점:**
- ✅ 문서 자동 저장 (questions.md)
- ✅ 에디터에서 편한 작성 (터미널 불편함 해소)
- ✅ 가독성 좋음
- ✅ 2W1H 프레임워크 강제

**AI는 대기 중:**
```
📝 질문지를 생성했습니다: problems/[문제명]/questions.md

에디터에서 편하게 작성하신 후, 다음과 같이 말씀해주세요:
"질문지 작성 완료했어. 이제 브레인스토밍 계속하자."
```

### 2.5. questions.md 읽기 및 분석

**사용자가 "질문지 작성 완료" 신호를 보내면:**

1. **questions.md 읽기**
2. **What/Why 재정의**
   - 모호한 용어 발견 시 추가 질문
   - "대규모 데이터" → "경쟁 상태? 빅데이터? 조회 최적화?"
   - "성능 개선" → "처리 속도? 응답 시간? 처리량?"

3. **산업/도메인 리서치 필요 여부 판단**
   - 사용자가 산업/도메인을 잘 모를 경우 **딥 리서치 프롬프트** 제안:

```
📋 [주제] 딥 리서치 프롬프트

다음 내용을 ChatGPT/Claude/Perplexity에 붙여넣고 리서치하세요:

---
[상황 설명]
나는 [목적]을 위해 [문제]를 해결하려고 합니다.

[리서치 질문]
1. [타겟 회사/산업]에서 "[키워드]"의 실제 의미는 무엇인가요?
2. 관련 기술 블로그 5개를 찾아 공통적으로 다루는 문제를 정리해주세요.
3. 채용 공고(JD) 10개를 분석해서 자주 등장하는 기술 키워드를 추출해주세요.
4. 1-2달 안에 완성 가능한 PoC 수준의 프로젝트 사례를 찾아주세요.

[추가 질문]
- 이 문제를 해결하는 오픈소스 프로젝트가 있나요?
- 실무에서 실제로 사용하는 솔루션은 무엇인가요?
---

💡 리서치 결과를 가지고 다시 브레인스토밍을 계속하겠습니다.
```

### 2.7. 사례 연구 리서치 (Case Study Research)

**What/Why가 명확해진 후, How로 넘어가기 전에 "선배들의 경험"을 수집합니다.**

**왜 필요한가:**
- 내 지식만으로는 "언제/왜 이 기술을 도입하는가"의 맥락이 부족
- 기술 블로그의 사례 연구에는 **도입 배경, 고민, 트레이드오프, 실패 경험**이 담겨 있음
- 이 인사이트가 `/1h-agile-phase`에서 **다이어그램 설계와 평가 지표 선정**에 직접 영향

**사례 연구 필요 여부 판단:**

| 상황 | 필요? | 이유 |
|------|-------|------|
| 처음 다루는 기술 (EDA, MSA 등) | ✅ 필수 | 도입 맥락을 모르면 PoC 범위 산정이 가설에 의존 |
| 익숙한 기술의 새로운 적용 | ⚠️ 선택 | 내 경험으로 부족한 부분만 보완 |
| 이미 리서치를 충분히 한 경우 | ❌ 불필요 | 중복 방지 |

**사례 연구 프롬프트 템플릿:**

사용자에게 다음 프롬프트를 제안합니다:

```
📋 [기술명] 사례 연구 리서치 프롬프트

다음 내용을 ChatGPT/Claude/Perplexity에 붙여넣고 리서치하세요:

---

[상황 설명]
나는 [목적]을 위해 [기술]을 도입하려고 합니다.
도입 전에 실제 사례를 통해 "언제/왜 도입했는가"를 이해하고 싶습니다.

[사례 연구 질문]
1. **도입 배경:** [기술]을 도입한 기업 사례 5개를 찾아주세요.
   - 도입 전 어떤 문제가 있었는가?
   - 왜 이 기술을 선택했는가? (다른 대안은?)
   - 도입 규모는? (전체 시스템? 일부 서비스?)

2. **트레이드오프:** 각 사례에서 도입 후 겪은 어려움은?
   - 예상치 못한 복잡성?
   - 운영 비용 증가?
   - 팀 학습 곡선?

3. **핵심 인사이트:** 사례들의 공통 패턴은?
   - "이런 상황에서 도입하면 효과적이다"
   - "이런 상황에서는 도입하지 마라"
   - "도입 시 반드시 함께 고려할 것 (예: 모니터링, DLQ)"

4. **PoC 관점:** 기술의 핵심 가치를 증명하려면 최소한 무엇을 구현해야 하는가?
   - 사례에서 가장 임팩트 있었던 시나리오는?
   - PoC에서 반드시 포함해야 할 테스트 시나리오는?

[출력 형식]
- 사례 요약 (회사명, 도입 배경, 결과)
- 공통 도입 조건 (이럴 때 도입)
- 공통 트레이드오프 (이건 감수해야 함)
- PoC 핵심 시나리오 추천
---

💡 리서치 결과를 가지고 다시 브레인스토밍을 계속하겠습니다.
```

**리서치 결과 분석:**

사용자가 사례 연구 결과를 공유하면:

1. **공통 도입 조건 추출** → What/Why 재검증
   - "우리 상황이 사례의 도입 조건과 일치하는가?"
   - 일치하지 않으면 → 범위/방향 재정의

2. **핵심 인사이트를 `2w-brainstorm.md`에 기록**
   - 도입 배경에서 발견한 패턴
   - 트레이드오프에서 발견한 주의점
   - PoC에 반드시 포함할 시나리오

3. **`/1h-agile-phase`에 전달할 인사이트 정리**
   - 다이어그램에 반영할 아키텍처 패턴
   - 평가 지표에 반영할 측정 항목
   - In/Out/Deferred 범위에 반영할 사항

4. **사례를 의사결정 근거로 직접 활용**

   > 사례 연구는 단순한 참고 자료가 아니라 **기술 선택과 범위 결정의 직접적 근거**가 된다.
   > "왜 이렇게 결정했는가?"에 대해 "X 기업이 Y 상황에서 Z 결과를 얻었다"로 답할 수 있어야 한다.

   - **기술 선택 근거**: 사례의 성공/실패가 기술 채택/제외의 이유가 됨
   - **범위 제외 근거**: 사례에서 전담팀/높은 학습 비용이 필요했다면 PoC 범위 밖으로 판단
   - **지표 선정 근거**: 사례에서 실제로 측정한 지표와 방식을 PoC 평가 지표에 채택
   - **ADR 근거**: 사례의 트레이드오프가 ADR(Architecture Decision Records)의 Why에 직접 인용됨

   **실제 예시 (EDA PoC):**
   | 사례 근거 | 의사결정 |
   |----------|---------|
   | 배민이 Exactly Once를 안정성 위해 제거 | → At Least Once + 멱등성 선택 (ADR) |
   | 배민도 Debezium에 전담팀 필요 | → Kafka Connect는 PoC 범위 밖 (Out of Scope) |
   | 올리브영이 DB 호출 86% 감소를 측정 | → Before/After 비교 방식으로 지표 설계 |
   | 쿠팡의 Cascading Failure 문제 | → 장애 격리 테스트를 핵심 검증 항목에 포함 |

**실제 예시 (EDA PoC):**

> **사례 연구 인사이트:**
> - 배민, 토스 등에서 EDA 도입 시 **장애 격리**가 핵심 동기
> - 도입 후 가장 큰 어려움: **비동기 이벤트 추적 (디버깅)**
> - PoC 핵심 시나리오: **서비스 장애 시 이벤트 큐잉 + 복구 후 처리**
>
> **→ how-diagram.md 반영:**
> - "장애 격리 테스트"를 핵심 평가 지표에 추가
> - "디버깅 어려움"을 트레이드오프로 기록 → 다음 PoC(모니터링) 연결

---

### 2.8. 안티패턴/베스트 프랙티스 리서치 (Anti-pattern / Best Practice Research)

**What/Why가 명확해지고 사례 연구가 완료된 후, How 설계 전에 "실무에서 자주 틀리는 것과 검증된 올바른 방식"을 수집합니다.**

**언제 필요한가:**

| 상황 | 필요? | 이유 |
|------|-------|------|
| 처음 다루는 기술 (EDA, MSA 등) | ✅ 필수 | 흔한 실수를 모르면 PoC에서 그 실수를 반복할 수 있음 |
| 면접용 포트폴리오 프로젝트 | ✅ 필수 | "EDA 도입 시 주의할 점은?" 질문에 답할 수 있어야 함 |
| 트레이드오프 문서 작성 예정 | ✅ 필수 | "왜 이렇게 했는가"에 안티패턴 회피 근거가 필요 |
| 이미 안티패턴을 충분히 아는 경우 | ❌ 불필요 | 중복 방지 |

**리서치 프롬프트 템플릿:**

사용자에게 다음 프롬프트를 제안합니다:

```
📋 [기술명] 안티패턴/베스트 프랙티스 리서치 프롬프트

다음 내용을 ChatGPT/Claude/Perplexity에 붙여넣고 리서치하세요:

---

[상황 설명]
나는 [목적]을 위해 [기술]을 학습하고 있습니다.
실무에서 자주 발생하는 실수(안티패턴)와 검증된 올바른 방식(베스트 프랙티스)을 파악하고 싶습니다.

[리서치 질문]
1. **안티패턴 (실무에서 자주 틀리는 것):**
   - [기술] 도입 시 가장 흔한 실수 5가지는?
   - 각 실수가 왜 문제인지, 어떤 증상으로 나타나는지?
   - PoC/초기 구현 단계에서 특히 조심해야 할 패턴은?

2. **베스트 프랙티스 (검증된 올바른 방식):**
   - 실무에서 검증된 설계 원칙 5가지는?
   - 각 원칙이 왜 중요한지, 어떤 문제를 예방하는지?
   - 초보자가 놓치기 쉬운 운영 관점의 필수 고려 사항은?

3. **포트폴리오/면접 관점:**
   - "이 기술 도입 시 주의할 점은?" 질문에 답할 수 있는 핵심 포인트는?
   - 내 PoC 코드가 안티패턴을 피하고 있음을 보여줄 수 있는 근거는?

[출력 형식]
- 안티패턴 목록 (이름, 증상, 왜 문제인가)
- 베스트 프랙티스 목록 (원칙, 이유, 적용 방법)
- 면접 답변 핵심 포인트
---

💡 리서치 결과를 가지고 다시 브레인스토밍을 계속하겠습니다.
```

**결과 분석 및 `2w-brainstorm.md` 기록:**

사용자가 안티패턴/BP 리서치 결과를 공유하면:

1. **PoC 코드와 대조** — "현재 설계가 어떤 안티패턴을 회피하고 있는가?"
   - 회피 중인 안티패턴 → 트레이드오프 문서에서 "왜 이렇게 했는가"의 근거로 활용
   - 미처 고려 못한 안티패턴 → Phase 완료 기준에 추가 또는 Out of Scope로 명시

2. **`2w-brainstorm.md`에 기록** — 안티패턴/BP 섹션 추가:

```markdown
## 안티패턴/베스트 프랙티스 (Anti-pattern / Best Practice)

### 안티패턴 (피해야 할 패턴)
- [안티패턴 1]: [증상] → [이 PoC에서의 대응]
- [안티패턴 2]: ...

### 베스트 프랙티스 (따라야 할 원칙)
- [원칙 1]: [이유] → [이 PoC에서의 적용 여부]
- [원칙 2]: ...

### `/1h-agile-phase` 반영 사항
- 완료 기준 추가: [안티패턴 회피 근거 문서화]
- 트레이드오프 문서 연결: [어떤 결정이 어떤 안티패턴을 피하는가]
```

**이후 Phase 설계 시 활용:**

안티패턴/BP 조사 결과는 다음과 같이 직접 활용된다:

| 활용처 | 내용 |
|--------|------|
| **트레이드오프 문서** | "왜 X 방식을 선택했는가" → Y 안티패턴을 회피하기 위함 |
| **ADR 근거** | ADR의 Why 섹션에 BP/안티패턴을 직접 인용 |
| **실무 도입 전략 문서** | 안티패턴 회피 + BP 적용이 실무 도입 시 주의 사항으로 연결 |
| **면접 대비** | "EDA 도입 시 주의할 점은?" → 조사한 안티패턴으로 구체적 답변 가능 |

**실제 예시 (EDA PoC):**

| 안티패턴/BP | 이 PoC에서의 대응 |
|------------|----------------|
| DLQ 없는 무한 재시도 → 파이프라인 정지 | DLQ 구현으로 실패 메시지 격리 (Phase 3 검증 항목) |
| Consumer가 단일 파티션을 멀티스레드 처리 → 순서 파괴 | 파티션 단위 순서 보장 + 멱등성으로 대응 |
| Kafka-as-DB → full-scan 성능 저하 | 이벤트는 상태 변경 통보 용도로만 사용, 조회는 DB 직접 |
| 이벤트에 과도한 데이터 포함 → 토픽 비대화 | OrderCompleted 이벤트에 필수 필드만 포함 (Fact 중심 설계) |

### 3. 1H로 넘어가기 전 체크 (⚠️ 중요!)

```
[범위 산정 매트릭스 참조]
- 기술 증명 (1-2달) → PoC
- 학습 (1-3달) → Tutorial
- 이직용 (1-2달) → PoC + 블로그
- 실제 서비스 (3-6달+) → MVP
```

#### 완성 가능성 검증
```
Q: "이 프로젝트의 '완성'을 어떻게 정의하시나요?"

[완성의 정의]
✅ 정량 지표 측정 (TPS, Latency, Before/After)
✅ 재현 가능 (README만 보고 실행 가능)
✅ 문서화/블로그 (Why/What/How 명확)
✅ 명확한 결론 ("어떤 상황에 어떤 방법")
```

- 처음 생각을 수정하기 시작
- 문제를 재정의하기 시작

## 역할 범위 (중요!)

### ✅ AI가 담당하는 것
- What/Why 질문 계속 던지기
- 모호한 용어 해체하도록 유도
- How 함정 감지 및 중단
- **패턴 감지 및 즉시 개입** (아래 패턴 감지 섹션 참조)
- 브레인스토밍 내용 문서화

### ❌ AI가 하지 않는 것

- How부터 이야기하지 마라
- 사용자의 첫 접근을 그대로 받아들이지 마라

## 패턴 감지 및 즉시 개입 (⚠️ 자동 탐지)

브레인스토밍 중 다음 패턴이 감지되면 **즉시 중단하고 질문**:

### 🔴 패턴 1: 제약 조건 무시

**신호:**
- 시간 제약을 묻지 않고 바로 기술 스택 논의
- "1-2달"이라고 했는데 3개월 이상 걸릴 프로젝트 계획
- "혼자"라고 했는데 팀 협업 전제의 아키텍처 제안

**즉시 개입:**
```
⚠️ 잠깐만요!

제약 조건을 확인해야 합니다:
- 시간: 얼마나 시간을 쓸 수 있나요?
- 협업: 혼자인가요? 팀인가요?
- 완성도: PoC? MVP? Production?

[범위 산정 매트릭스]를 참조해서 현실적인 범위를 다시 정의하겠습니다.
```

### 🔴 패턴 2: 가설 의존 (산업 리서치 없음)

**신호:**
- "대규모 트래픽 = MSA"처럼 검증되지 않은 가정
- 타겟 회사/산업의 실제 니즈를 확인하지 않음
- 기술 블로그, JD 분석 없이 기술 선택

**즉시 개입:**
```
⚠️ 잠깐만요!

이것은 검증된 정보인가요, 가설인가요?

[딥 리서치 프롬프트]를 드릴게요. 다음 내용을 리서치한 후 다시 브레인스토밍을 계속하겠습니다:

1. [타겟 산업]에서 "[키워드]"의 실제 의미
2. 관련 기술 블로그 5개 분석
3. JD 10개에서 키워드 추출
4. 1-2달 완성 가능한 PoC 사례
```

### 🔴 패턴 3: 프로젝트 형태 혼동

**신호:**
- "학습/이직"이 목적인데 Production 수준 설계
- "PoC"라고 했는데 3개 도메인, 헥사고날, Kafka 등 포함
- "토이 프로젝트"인데 MVP 수준 요구사항

**즉시 개입:**
```
⚠️ 잠깐만요!

목적과 형태가 일치하지 않습니다:

- 목적: [이직/학습/비즈니스]
- 현재 계획: [MVP/PoC 수준 판단]
- 권장 형태: [PoC/Tutorial/MVP]

[완성의 정의]를 참조해서 범위를 다시 조정하겠습니다.
```

### 🔴 패턴 4: 사례 연구 없이 How 진입

**신호:**
- 처음 다루는 기술인데 "이렇게 하면 되지 않을까?" (자기 가설만으로 진행)
- 기술의 도입 배경/트레이드오프를 모르는 채 구현 방향 논의
- "왜 이 기술을 도입하는가?"에 대한 답이 추상적 (예: "요즘 다 쓰니까")

**즉시 개입:**
```
⚠️ 잠깐만요!

이 기술을 처음 다루신다면, 선배들의 경험을 먼저 참고하는 게 좋겠습니다.

[사례 연구 프롬프트]를 드릴게요. 다음 내용을 리서치한 후 다시 브레인스토밍을 계속하겠습니다:

1. 실제 기업에서 [기술]을 도입한 배경과 결과
2. 도입 후 겪은 트레이드오프
3. PoC에서 반드시 포함해야 할 핵심 시나리오

💡 이 인사이트가 `/1h-agile-phase`에서 다이어그램과 평가 지표 설계에 직접 반영됩니다.
```

### 📊 패턴 감지 체크리스트

브레인스토밍 중 지속적으로 확인:
- [ ] 제약 조건 (시간/협업/완성도) 물었는가?
- [ ] 산업/타겟 리서치 제안했는가?
- [ ] **사례 연구 인사이트** 수집했는가? (또는 불필요 판단했는가?)
- [ ] 프로젝트 형태가 목적과 일치하는가?
- [ ] 범위가 완성 가능한가?

**3개 이상 체크되지 않으면 How로 넘어가지 마라!**

## Success Criteria

- [ ] **제약 조건 확인** (시간/협업/완성도)
- [ ] **범위 산정 완료** (PoC/MVP/Tutorial 중 선택)
- [ ] **완성 가능성 검증** (제약 내 완성 가능)
- [ ] 브레인스토밍 내용이 파일로 기록됨
- [ ] **`/1h-agile-phase` 실행 안내 완료**

### 선택 (Nice to Have)
- [ ] 산업 리서치 프롬프트 제공 (필요시)
- [ ] **사례 연구 프롬프트 제공** (처음 다루는 기술일 때)
- [ ] 사례 연구 인사이트가 `/1h-agile-phase`에 전달할 형태로 정리됨
- [ ] 패턴 감지 및 개입 (제약 무시/가설 의존/형태 혼동/사례 연구 미수행)

### 최종 목표
- [ ] 사용자가 "아, 이게 진짜 문제였구나" 깨달음
- [ ] **2W 명확화 → `/1h-agile-phase`으로 자연스러운 전환**
