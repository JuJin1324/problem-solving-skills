# 문제 해결 (Solve Problem)

## 목적

정의된 문제를 **1-3단계로 나눠 해결**하고, 끝나면 **배운 점을 기록**합니다.

> "문제 해결의 끝은 해결 자체가 아니라, 그 과정에서 무엇을 배웠는지다."

## 핵심 철학

- 모든 문제 해결은 **문제 정의**에서 시작하고 **문제 해결**에서 끝난다
- 해결 단계는 **최대 3개** — 더 필요하면 문제를 쪼개라
- 해결 후 반드시 **"뭘 배웠는가"** 되돌아보기
- 한 루프로 안 풀리면 **재정의 → 재해결** — 이것이 자연스러운 성장

## 전체 흐름

```
/define-problem → problem.md
       ↓
/solve-problem → 1-3단계 해결 → solution.md ← 현재 스킬
       ↓
   (미해결?) → /define-problem (재정의) → /solve-problem → ...
```

---

## 작동 방식

### 1단계. 문제 정의 확인

`problems/[문제명]/problem.md`를 읽는다.

**파일이 없으면:**
```
⚠️ 문제 정의가 없습니다.

먼저 `/define-problem`을 실행해서 문제를 정의해주세요.
```

**파일이 있으면** 핵심을 요약해서 확인:
```
📋 문제 정의 확인:
- What: [무엇]
- Why: [왜]
- Done: [완료 기준]

이 문제를 해결하겠습니다. 맞나요?
```

**problem.md 없이 사용자가 직접 문제를 말한 경우:**
- 최소한 What/Why/Done을 대화에서 확인한 후 진행
- 가능하면 problem.md 작성을 권장하되, 강요하지 않음

### 2단계. 해결 단계 설계 (최대 3단계)

문제를 해결하기 위한 **구체적인 단계를 1-3개**로 나눈다:

```
🔧 해결 단계:

1단계: [구체적 행동]
   → 산출물: [무엇이 만들어지는가]

2단계: [구체적 행동]
   → 산출물: [무엇이 만들어지는가]

3단계: [구체적 행동]
   → 산출물: [무엇이 만들어지는가]
```

**단계 설계 원칙:**
- 각 단계는 **구체적 행동**과 **확인 가능한 산출물**을 포함
- **3단계를 넘기면** 문제가 너무 큰 것 → 문제를 쪼개라고 제안
- 사용자 확인 후 실행 시작
- 단계 수는 문제 크기에 맞춤 (1단계로 끝나면 1단계로 OK)

### 3단계. 단계별 실행

각 단계를 **하나씩** 실행한다:

```
▶️ 1단계 시작: [행동]
  ... (실행) ...
✅ 1단계 완료: [산출물 확인]

▶️ 2단계 시작: [행동]
  ... (실행) ...
✅ 2단계 완료: [산출물 확인]
```

**실행 중 원칙:**
- 한 단계를 끝내고 다음으로 넘어감 (동시에 여러 단계 진행 ❌)
- 막히면 사용자와 상의 (무작정 돌파 ❌)
- 범위가 커지면 멈추고 "이건 다음 루프에서" 제안
- 예상치 못한 문제 발견 시 기록해두고, 현재 단계에 집중

### 4단계. 완료 확인

problem.md의 Done 기준과 대조:
```
📊 완료 기준 확인:
- [Done 기준 1]: ✅ 달성 / ❌ 미달성
- [Done 기준 2]: ✅ 달성 / ❌ 미달성
```

### 5단계. 배운 점 기록 (회고)

사용자에게 묻는다:
```
🎓 이 문제를 풀면서 배운 게 있나요?

예시:
- "처음에 A로 접근했는데 B가 더 나았다"
- "X 개념을 이번에 처음 이해했다"
- "생각보다 Y가 핵심이었다"

짧아도 괜찮습니다. 한 줄이라도!
```

사용자가 말하기 어려워하면 AI가 관찰한 것을 초안으로 제안할 수 있다.
단, 최종 내용은 사용자가 확인/수정한다.

### 6단계. solution.md 작성

**파일 경로:** `problems/[문제명]/solution.md`

```markdown
# [문제명] - 해결

📅 날짜: [날짜]

## 문제 요약

[problem.md의 문제 정의 1문장]

## 해결 단계

### 1단계: [행동]
[무엇을 했는지, 결과]

### 2단계: [행동]
[무엇을 했는지, 결과]

### 3단계: [행동]
[무엇을 했는지, 결과]

## 완료 확인

- [Done 기준]: ✅/❌

## 배운 점

[사용자가 공유한 배운 점]

---

> 문제가 완전히 해결되지 않았다면 `/define-problem`으로 문제를 재정의하세요.
> 더 촘촘한 단계가 필요하다면 `/sprint-start`를 시도하세요.
```

### 7단계. 다음 단계 안내

**완전 해결된 경우:**
```
✅ 문제가 해결되었습니다!

📄 파일:
- 문제 정의: problems/[문제명]/problem.md
- 해결 과정: problems/[문제명]/solution.md

🎓 배운 점: [핵심 1줄 요약]
```

**미해결 또는 새로운 문제 발견된 경우:**
```
⚠️ 아직 완전히 해결되지 않았습니다.

남은 부분:
- [미해결 항목]

🔄 `/define-problem`을 다시 실행해서 남은 문제를 재정의하세요.
   (이것이 자연스러운 루프입니다!)
```

## 역할 범위

### AI가 하는 것
- problem.md 읽고 문제 확인
- 해결 단계 1-3개 제안 (사용자 확인 필수)
- 단계별 실행 지원
- Done 기준 대조
- 배운 점 질문
- solution.md 작성

### AI가 하지 않는 것
- 문제 정의 없이 바로 해결 시작 ❌
- 4단계 이상으로 복잡하게 만들기 ❌
- 사용자 확인 없이 단계 실행 ❌
- 배운 점을 사용자 대신 확정하기 ❌ (초안 제안은 OK)
- 막혔을 때 무작정 돌파 ❌

---

## 성장 경로

이 스킬에 익숙해지면, 더 체계적인 실행이 필요할 때:

| 필요한 것 | 스킬 | 추가되는 것 |
|-----------|------|------------|
| Sprint 단위 계획 | `/sprint-start` | 타임박스, US/Step 구조 |
| 진행 상황 추적 | `/sprint-status` | 완료율, 일정 상태 시각화 |
| 깊은 회고 | `/sprint-complete` | 정량 평가, Action Items |

**정의 → 해결** 루프의 본질은 같다. 단계가 더 촘촘해질 뿐이다.
