# 문제 해결 (Solve Problem)

## 목적

정의된 문제에 **맞는 방법론을 선택**하고, 해당 방법론으로 **해결 후 배운 점을 기록**합니다.

> "문제 해결의 끝은 해결 자체가 아니라, 그 과정에서 무엇을 배웠는지다."

## 핵심 철학

- 모든 문제 해결은 **문제 정의**에서 시작하고 **문제 해결**에서 끝난다
- 그 사이의 과정(설계, 리서치, 스프린트 등)은 **플러그인**이다
- 어떤 플러그인을 끼울지가 곧 **방법론 선택**이다
- 한 루프로 안 풀리면 **재정의 → 재해결** — 이것이 자연스러운 성장

## 전체 흐름

```
/define-problem → problem.md
       ↓
/solve-problem → 방법론 선택 → [플러그인] → 해결 → solution.md
       ↓
   (미해결?) → /define-problem (재정의) → /solve-problem → ...
```

---

## 방법론과 플러그인

### 구조

```
define-problem ──→ [ 플러그인 ] ──→ solve-problem
                       ↑
                  방법론이 결정함
```

기존 스킬들은 이 사이에 끼우는 **플러그인**이다:

| 플러그인 | 역할 | 해당 스킬 |
|----------|------|----------|
| 리서치 | 사례/안티패턴 조사 | `/2w-brainstorm` (deep research) |
| 설계 | 범위/구조 시각화 | `/1h-agile-phase` |
| 실행관리 | Sprint 계획/추적/회고 | `/sprint-start` `/sprint-status` `/sprint-complete` |
| 기록 | 의사결정 아카이브 | `/record-adr` |

### 방법론 선택지

| 방법론 | 언제 쓸까 | 플러그인 | 예상 소요 |
|--------|----------|---------|----------|
| **직행** | 간단, 명확, 바로 할 수 있음 | (없음) | 분~시간 |
| **설계 후 실행** | 방향이 여러 개, 범위 통제 필요 | +설계 | 시간~반나절 |
| **스프린트** | 며칠 이상, 진행 추적 필요 | +설계 +실행관리 | 일~주 |
| **풀 엔지니어링** | 복잡한 시스템, 리스크 관리 필요 | +리서치 +설계 +실행관리 +기록 | 주~ |

## 작동 방식

### 1단계. 문제 정의 확인

`problems/[문제명]/problem.md`를 읽는다.

**파일이 없으면:**
```
⚠️ 문제 정의가 없습니다.

먼저 `/define-problem`을 실행해서 문제를 정의해주세요.
```

**파일이 있으면** 핵심을 요약해서 확인:
```
📋 문제 정의 확인:
- What: [무엇]
- Why: [왜]
- Done: [완료 기준]

이 문제를 해결하겠습니다. 맞나요?
```

**problem.md 없이 사용자가 직접 문제를 말한 경우:**
- 최소한 What/Why/Done을 대화에서 확인한 후 진행
- 가능하면 problem.md 작성을 권장하되, 강요하지 않음

### 2단계. 방법론 선택

문제의 특성을 보고 AI가 **방법론을 추천**한다. 사용자가 최종 선택.

```
🔍 이 문제에 맞는 방법론을 골라볼까요?

1️⃣ 직행 — 바로 1-3단계로 해결
   → 문제가 명확하고 바로 시작할 수 있을 때

2️⃣ 설계 후 실행 — 범위/구조를 먼저 그린 후 해결
   → 방향이 여러 개이거나 범위 통제가 필요할 때
   → 플러그인: /1h-agile-phase

3️⃣ 스프린트 — Sprint 단위로 계획하고 추적하며 해결
   → 며칠 이상 걸리거나 진행 관리가 필요할 때
   → 플러그인: /1h-agile-phase → /sprint-start → /sprint-status → /sprint-complete

4️⃣ 풀 엔지니어링 — 리서치부터 설계, 실행, 기록까지 전부
   → 복잡한 시스템이거나 리스크 관리가 필요할 때
   → 플러그인: /2w-brainstorm → /1h-agile-phase → /sprint-start → ... → /record-adr

💡 AI 추천: [N번 - 이유 1줄]
```

**방법론 추천 기준:**
- 완료 기준이 단순하고 예상 단계가 1-3개 → 직행
- 방향이 여러 개이거나 구현 전 구조 파악 필요 → 설계 후 실행
- "~일 안에", "여러 기능", 진행 추적 필요 → 스프린트
- "시스템 설계", "기술 선택 필요", 리스크 언급 → 풀 엔지니어링
- **확신이 없으면 직행 추천** (시작 후 필요하면 올릴 수 있다)

### 3단계. 방법론별 실행

#### 직행 선택 시

바로 해결 단계 설계로 넘어간다 (4단계).

#### 설계 후 실행 선택 시

```
📐 먼저 범위와 구조를 잡겠습니다.

→ `/1h-agile-phase` 실행
→ 완료 후 이 스킬로 돌아와 해결 단계를 설계합니다.
```

`/1h-agile-phase` 완료 후 4단계로 진행.

#### 스프린트 선택 시

```
🏃 Sprint로 관리하며 해결하겠습니다.

→ `/1h-agile-phase` 실행 (범위/Phase 설계)
→ `/sprint-start` 실행 (Sprint 계획)
→ Sprint 실행 중 `/sprint-status`로 추적
→ 완료 시 `/sprint-complete`로 회고
→ 이후 이 스킬로 돌아와 최종 정리합니다.
```

Sprint 완료 후 7단계(배운 점)로 진행.

#### 풀 엔지니어링 선택 시

```
🔬 체계적으로 리서치부터 시작하겠습니다.

→ `/2w-brainstorm` 실행 (깊은 리서치 포함)
→ `/1h-agile-phase` 실행 (범위/Phase 설계)
→ `/sprint-start` 실행 (Sprint 계획)
→ Sprint 실행 중 의사결정 시 `/record-adr`
→ 완료 시 `/sprint-complete`로 회고
→ 이후 이 스킬로 돌아와 최종 정리합니다.
```

Sprint 완료 후 7단계(배운 점)로 진행.

### 4단계. 해결 단계 설계 (직행/설계 후 실행)

문제를 해결하기 위한 **구체적인 단계를 1-3개**로 나눈다:

```
🔧 해결 단계:

1단계: [구체적 행동]
   → 산출물: [무엇이 만들어지는가]

2단계: [구체적 행동]
   → 산출물: [무엇이 만들어지는가]

3단계: [구체적 행동]
   → 산출물: [무엇이 만들어지는가]
```

**단계 설계 원칙:**
- 각 단계는 **구체적 행동**과 **확인 가능한 산출물**을 포함
- **3단계를 넘기면** 문제가 너무 큰 것 → 방법론 업그레이드 제안
- 사용자 확인 후 실행 시작
- 단계 수는 문제 크기에 맞춤 (1단계로 끝나면 1단계로 OK)

### 5단계. 단계별 실행

각 단계를 **하나씩** 실행한다:

```
▶️ 1단계 시작: [행동]
  ... (실행) ...
✅ 1단계 완료: [산출물 확인]

▶️ 2단계 시작: [행동]
  ... (실행) ...
✅ 2단계 완료: [산출물 확인]
```

**실행 중 원칙:**
- 한 단계를 끝내고 다음으로 넘어감 (동시에 여러 단계 진행 ❌)
- 막히면 사용자와 상의 (무작정 돌파 ❌)
- 범위가 커지면 멈추고 "이건 다음 루프에서" 제안
- 예상치 못한 문제 발견 시 기록해두고, 현재 단계에 집중

### 6단계. 완료 확인

problem.md의 Done 기준과 대조:
```
📊 완료 기준 확인:
- [Done 기준 1]: ✅ 달성 / ❌ 미달성
- [Done 기준 2]: ✅ 달성 / ❌ 미달성
```

### 7단계. 배운 점 기록 (회고)

사용자에게 묻는다:
```
🎓 이 문제를 풀면서 배운 게 있나요?

예시:
- "처음에 A로 접근했는데 B가 더 나았다"
- "X 개념을 이번에 처음 이해했다"
- "생각보다 Y가 핵심이었다"
- "직행으로 시작했는데 스프린트가 필요했다" ← 방법론 피드백도 OK

짧아도 괜찮습니다. 한 줄이라도!
```

사용자가 말하기 어려워하면 AI가 관찰한 것을 초안으로 제안할 수 있다.
단, 최종 내용은 사용자가 확인/수정한다.

### 8단계. solution.md 작성

**파일 경로:** `problems/[문제명]/solution.md`

```markdown
# [문제명] - 해결

📅 날짜: [날짜]
🔧 방법론: [직행 / 설계 후 실행 / 스프린트 / 풀 엔지니어링]

## 문제 요약

[problem.md의 문제 정의 1문장]

## 사용한 방법론

[선택한 방법론과 이유 1줄]
[사용한 플러그인 목록]

## 해결 단계

### 1단계: [행동]
[무엇을 했는지, 결과]

### 2단계: [행동]
[무엇을 했는지, 결과]

### 3단계: [행동]
[무엇을 했는지, 결과]

## 완료 확인

- [Done 기준]: ✅/❌

## 배운 점

[사용자가 공유한 배운 점]

---

> 문제가 완전히 해결되지 않았다면 `/define-problem`으로 문제를 재정의하세요.
```

### 9단계. 다음 단계 안내

**완전 해결된 경우:**
```
✅ 문제가 해결되었습니다!

📄 파일:
- 문제 정의: problems/[문제명]/problem.md
- 해결 과정: problems/[문제명]/solution.md

🔧 사용한 방법론: [방법론명]
🎓 배운 점: [핵심 1줄 요약]
```

**미해결 또는 새로운 문제 발견된 경우:**
```
⚠️ 아직 완전히 해결되지 않았습니다.

남은 부분:
- [미해결 항목]

🔄 `/define-problem`을 다시 실행해서 남은 문제를 재정의하세요.
   (이것이 자연스러운 루프입니다!)

💡 다음 루프에서는 방법론을 올려볼까요?
   직행 → 설계 후 실행 → 스프린트 → 풀 엔지니어링
```

## 역할 범위

### AI가 하는 것
- problem.md 읽고 문제 확인
- 문제 특성에 맞는 방법론 추천
- 방법론에 따라 플러그인 스킬 안내/실행
- 해결 단계 1-3개 제안 (직행/설계 후 실행 시)
- 단계별 실행 지원
- Done 기준 대조
- 배운 점 질문
- solution.md 작성

### AI가 하지 않는 것
- 문제 정의 없이 바로 해결 시작 ❌
- 방법론 선택 없이 바로 플러그인 실행 ❌
- 4단계 이상으로 복잡하게 만들기 ❌ (방법론 업그레이드 제안)
- 사용자 확인 없이 단계 실행 ❌
- 배운 점을 사용자 대신 확정하기 ❌ (초안 제안은 OK)
- 막혔을 때 무작정 돌파 ❌

---

## 안티패턴

- 문제가 간단한데 풀 엔지니어링을 권유 → 오버엔지니어링
- 직행으로 시작했는데 막혀도 방법론 변경을 제안하지 않음
- 플러그인 스킬을 실행하면서 solve-problem의 흐름을 잃어버림
- 방법론 선택에서 너무 오래 고민하게 함 (확신 없으면 직행 후 올리기)
