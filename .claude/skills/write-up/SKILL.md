# Write-up

## 목적
사용자의 질문에 대해 구조적으로 바라볼 수 있는 답변을 마크다운 문서로 남긴다.
정리되지 않은 생각이나 메모를 다른 사람(혹은 미래의 나)이 읽을 수 있는 문서로 바꾸는 것이 핵심이다.

이 스킬은 **초안 생성**과 **리뷰/고도화**를 모두 담당한다.
- **초안 생성**: 사용자의 메모/생각을 구조화된 문서로 만든다.
- **리뷰/고도화**: 초안을 함께 보면서 사용자가 의견을 덧붙이거나, AI에게 의견을 물어 문서를 발전시킨다.

모든 판단에서 **독자 이해**를 최우선으로 한다. 이를 위한 핵심 기준:
- **시각화**: 전체 흐름을 Mermaid 다이어그램으로 항상 먼저 보여준다. 독자가 큰 그림을 잡은 뒤 디테일로 내려가게 한다.
- **간소화 vs 문맥 보존**: 이 둘은 항상 긴장 관계에 있다. 간소화는 불필요한 부분을 줄여 인지 과부하를 낮추고, 문맥 보존은 "왜"와 배경을 남겨 독자의 이해를 돕는다. 충돌할 때는 **문맥 보존**을 선택한다. 산출물의 목적이 독자의 이해이므로, 짧지만 맥락이 끊긴 문서는 목적을 달성하지 못하기 때문이다.

## 출력 규칙
- 항상 마크다운(`.md`) 형식으로 출력한다.
- 저장 경로: `write-ups/` 디렉터리 (프로젝트 루트 기준)
- 파일명: 영문 케밥 케이스 (예: `collaboration-principles.md`)

## 작동 방식

### 1단계. 입력 분석
- 사용자가 제시한 내용(질문, 메모, 초안 등)을 읽고, 산출물의 **독자**와 **목적**을 파악한다.
- 명확하지 않으면 사용자에게 확인한다.

### 2단계. 전체 구조 다이어그램
- 내용의 큰 덩어리만 Mermaid 다이어그램으로 먼저 구성한다.
- 다이어그램 아래에 각 덩어리가 무엇을 말하는지 상세 설명을 붙인다.

### 3단계. 섹션별 다이어그램 + 상세 내용
- 각 섹션마다 해당 섹션의 흐름을 다이어그램으로 보여준다.
- 다이어그램 아래에 각 구성 요소를 텍스트로 상세 설명한다.
- 간소화 기준: 문서 내 불필요한 반복 제거, 불필요한 수식어 제거, 핵심 정보 유지. (여러 write-up을 리뷰할 때 문서 간에 비슷한 아이디어가 보일 수 있다. 이것은 제거 대상이 아니다. 같은 아이디어를 다른 맥락에서 반복해서 만나는 것은 독자에게 기억의 강화이기 때문이다.)
- 문맥 보존 기준: 간소화 후 "왜"가 사라진 부분은 연결 문장으로 보충한다.

### 4단계. 최종 검수
- 독자가 이 산출물만 읽고 내용을 이해할 수 있는가?
- 다이어그램이 전체 흐름을 정확히 반영하는가?
- 과도한 간소화로 맥락이 사라진 부분은 없는가?
- 다이어그램의 연결(순차/병렬)이 항목 간 실제 관계를 반영하는가?
- 상위 개념과 하위 항목의 성격이 일치하는가? (예: 핵심 교훈 아래 성격이 다른 운영 원칙이 섞여 있지 않은가)

## 안티패턴
- 원본을 그대로 복사해 구조만 바꾸기 (내용 정리 없이 형식만 변경)
- 과도한 간소화로 "왜"가 사라진 산출물
- 독자 수준을 고려하지 않은 전문 용어 나열
- 논리적 흐름 없이 항목만 나열
- 지식의 저주: 리뷰/고도화 시 채팅 문맥을 알고 있어서 부연설명 없이 결론만 산출물에 반영하는 것. 대화에 참여하지 않은 미래의 독자는 이해할 수 없다.
- 다이어그램에서 관계를 잘못 표현: 병렬 교훈을 순차 흐름으로, 또는 인과관계가 없는 항목을 인과로 연결하는 것.

## 완료 조건

- 전체 흐름 다이어그램이 포함됨
- 독자가 산출물만 읽고 내용을 이해할 수 있음
- 문맥이 끊기는 부분 없이 자연스럽게 읽힘
