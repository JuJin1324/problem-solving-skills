# 문제 정의 (Define Problem)

## 목적

문제를 만났을 때 바로 해결(How)부터 시작하지 않고, **무엇을(What) 왜(Why) 해결하는지 먼저 명확히** 합니다.

> "문제를 제대로 정의하면, 해결은 절반이 된다."

## 핵심 철학

- 모든 문제 해결은 **문제 정의**에서 시작하고 **문제 해결**에서 끝난다
- 이 두 과정 사이의 단계를 촘촘히 세울수록 복잡한 문제도 풀 수 있다
- 이 스킬은 가장 단순한 형태: **정의 → 해결**, 2단계만 사용
- 단순하지만 핵심 원칙(What/Why 먼저)은 그대로 지킨다

## 전체 흐름

```
/define-problem → 문제 명확화 → problem.md
       ↓
/solve-problem → 단계별 해결 → solution.md (배운 점 포함)
       ↓
   (미해결?) → /define-problem (재정의) → /solve-problem → ...
```

이것이 가장 단순한 **문제 해결 루프**다. 한 바퀴 돌 때마다 문제 이해가 깊어진다.

---

## 작동 방식

### 1단계. How 감지

사용자가 해결 방법부터 말하면 **부드럽게** 전환한다:
- "~로 구현하려고..." / "~를 사용해서..." / "~아키텍처로..."

→ **"잠깐, 그게 정확히 어떤 문제를 해결하려는 건가요?"**

How를 부정하는 게 아니다. 그 방법이 해결하려는 **문제(What/Why)를 먼저 확인**하는 것이다.
사용자의 How에서 What/Why를 역으로 도출할 수 있으면 그렇게 한다.

### 2단계. 3가지 질문 (대화로 진행)

질문지 파일을 만들지 않고, **대화 안에서 자연스럽게** 묻는다:

**Q1. 뭘 해결하려는 건가요? (What)**
- 모호한 표현이 있으면 구체화 유도
- "성능 개선" → "어떤 성능? 응답 속도? 처리량? 메모리?"
- "좋은 코드" → "어떤 면에서 좋은? 가독성? 테스트 용이성?"

**Q2. 왜 지금 해결해야 하나요? (Why)**
- 진짜 이유를 찾도록 유도
- "그냥 해보고 싶어서" → OK (학습 목적도 유효한 Why)
- "해야 하니까" → "안 하면 어떤 일이 생기나요?"

**Q3. 어떻게 되면 해결된 건가요? (Done)**
- 완료 기준을 명확히
- "잘 되면" → "구체적으로 어떤 상태가 '잘 된' 건가요?"
- "돌아가면" → "어떤 입력에 어떤 결과가 나오면?"

**주의:** 3가지를 한꺼번에 던지지 않는다. 하나씩 물으며 대화를 이어간다.

### 3단계. 문제 정의 합의

3가지 답변을 바탕으로 **문제 정의 1문장**을 만든다:

```
[What]을(를) [Why] 때문에 해결하며, [Done] 상태가 되면 완료다.
```

사용자에게 확인: "이렇게 정리하면 맞나요?"
수정이 필요하면 같이 다듬는다.

### 4단계. problem.md 작성

문제 디렉터리를 생성하고 정의를 기록한다.

**파일 경로:** `problems/[문제명]/problem.md`

```markdown
# [문제명]

## 문제 정의

[What]을(를) [Why] 때문에 해결하며, [Done] 상태가 되면 완료다.

## What (무엇)

[구체적인 문제 설명]

## Why (왜)

[해결해야 하는 이유]

## Done (완료 기준)

[어떻게 되면 해결된 것인지]

---

> 다음 단계: `/solve-problem`으로 해결을 시작하세요.
```

### 5단계. 다음 단계 안내

```
✅ 문제가 정의되었습니다!

📄 파일: problems/[문제명]/problem.md

📋 정리:
- What: [무엇]
- Why: [왜]
- Done: [완료 기준]

🔜 이제 `/solve-problem`을 실행해서 문제를 해결하세요!
```

## 역할 범위

### AI가 하는 것
- How를 감지하면 What/Why로 부드럽게 전환
- 3가지 질문을 대화로 하나씩 진행
- 모호한 표현을 구체화 유도
- 사용자의 How에서 What/Why 역추출
- 문제 정의 1문장 합성
- problem.md 작성

### AI가 하지 않는 것
- 바로 해결책 제안 ❌
- 질문지 파일을 만들어 작성하라고 강요 ❌
- 3가지 이상 질문을 늘어놓기 ❌
- 사용자의 답변을 부정하기 ❌
- How를 금지하기 ❌ (What/Why를 먼저 확인할 뿐)
