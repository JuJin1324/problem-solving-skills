# Code Style Conventions

## 공통 원칙
- 이름은 의도를 드러내야 한다.
- 함수는 한 가지 책임만 가진다.
- 중복은 추출하고, 추상화는 과도하지 않게 적용한다.

## 네이밍
- 클래스/타입: 명사형
- 함수/메서드: 동사형
- 불리언: `is/has/can` 접두 사용
- 축약어 최소화 (`amt` 대신 `amount`)

## 구조
- 하나의 파일/클래스에 서로 다른 변경 이유를 섞지 않는다.
- 공개 API는 작고 예측 가능하게 유지한다.
- 예외/오류 처리는 명시적으로 작성한다.

## 테스트
- 핵심 비즈니스 규칙 테스트 필수
- 실패/경계 케이스 테스트 필수
- 테스트명은 시나리오와 기대 결과를 포함한다.

## 리뷰 체크리스트
- [ ] 네이밍이 의도를 충분히 설명하는가
- [ ] 함수 길이와 복잡도가 과도하지 않은가
- [ ] 오류 처리와 로그가 누락되지 않았는가
- [ ] 테스트가 핵심 규칙을 커버하는가
- [ ] 불필요한 주석/죽은 코드가 없는가
