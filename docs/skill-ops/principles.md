# 문제 해결 프레임워크 원칙

## 목적
이 문서는 스킬 실행 전반에서 공통으로 지켜야 할 판단 원칙을 정의한다.
`framework.md`가 단계/라우팅/산출물 계약을 다룬다면, 본 문서는 실행 태도와 의사결정 기준을 다룬다.

## 적용 범위
1. `.codex/skills/`의 모든 Active 스킬
2. 사용자와의 상호작용 방식(질문, 선택지 제시, 리뷰 대기, 보고 방식)
3. 스킬별 판단 기준 정합성 점검

## Core (공통 강제)
아래 항목은 모든 스킬에 공통 적용한다.

1. 사고 확장 우선: 정답 전달보다 사용자 사고 확장을 우선한다.
2. 선택 보조 중심: 단일 답 강요 대신 대안과 트레이드오프를 제시한다.
3. 문서 중심 협업: 핵심 판단/증적/결정을 문서로 남겨 비동기 검토를 가능하게 한다.
4. 비강압 의사결정: 승인/보류/재검토를 정식 선택지로 유지하고 선택을 재촉하지 않는다.
5. 증적 기반 판단: 완료/상태/결정에는 확인 가능한 근거(실행 결과, 로그, 링크)를 남긴다.
6. 단일 소스 유지: 동일 사실(상태/리스크/결정)을 여러 문서에 중복 관리하지 않는다.
7. 범위 투명성: 필요한 경우 In/Out/Unknown을 분리해 범위 팽창을 방지한다.

## Modules (조건부 적용)
아래 항목은 스킬 성격에 따라 선택 적용한다. 각 `SKILL.md`에는 `Core + 적용 Module`만 명시한다.

### M-PIPELINE-CONTRACT
- 의미: `framework.md`의 단계 게이트/라우팅/산출 디렉터리 계약을 따른다.
- 적용 대상: Direction/Delivery/Learning 파이프라인을 직접 다루는 스킬

### M-WIP-ONE-US
- 의미: 기본 실행 단위를 US 1개로 제한하고 순차 루프를 유지한다.
- 적용 대상: Delivery 실행 스킬(설계/구현/테스트/상태관리)

### M-STEP-REVIEW-GATE
- 의미: 마이크로 스텝마다 구현 -> 검증 -> 사용자 리뷰 대기를 강제한다.
- 적용 대상: `execute-implementation` 계열 스킬

### M-SCHEDULE-VISIBILITY
- 의미: 스프린트 기간, US 목표일, 일정 상태(Ahead/On-track/Delayed)를 일관되게 관리한다.
- 적용 대상: 일정 기준을 생성/모니터링하는 스킬

### M-REPLAN-TRIGGER
- 의미: 리플랜은 사전 정의된 트리거(예: 일정 편차, blocker 발생)에서만 발동한다.
- 적용 대상: 상태 재평가/재계획 스킬

### M-EXPERIENCE-LIFECYCLE
- 의미: 경험 자산을 `seeded/validated/deprecated`로 관리하고 `validated`를 우선 활용한다.
- 적용 대상: 경험 자산 관리 스킬

## 운영 규칙
1. Core는 모든 스킬에 강제한다.
2. Module은 스킬 책임과 충돌하지 않는 범위에서만 적용한다.
3. 스킬은 Core를 중복 서술하지 않고 `적용 원칙(Core + Modules)`만 선언한다.
4. 스킬 원칙 충돌 시에는 형식적 우선순위보다 목적 정합성을 우선 판단하고, 반복 충돌이 발생하면 본 문서를 개정한다.
5. 파이프라인 계약 충돌 시 `framework.md`를 우선 적용한다.
